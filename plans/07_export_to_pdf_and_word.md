## 2. Export Architecture Decision

### Option A: Backend Export (Recommended)

The backend receives the study aid content, generates the document, and returns it as a file download.

```
Frontend                    Backend
   │                           │
   │  POST /api/export         │
   │  { content, format,       │
   │    title }                 │
   │  ─────────────────────►   │
   │                           │  Generate PDF/DOCX
   │                           │  using fpdf2 / python-docx
   │  ◄─────────────────────   │
   │  Binary file response     │
   │  (Content-Disposition:    │
   │   attachment)             │
   │                           │
   Browser downloads file      │
```

**Pros:** Full control over formatting, consistent output, works on all browsers.
**Cons:** Network round-trip, server resource usage.

### Option B: Frontend Export (Alternative)

Use JavaScript libraries to generate documents client-side.

**Pros:** No server round-trip, works offline.
**Cons:** Limited formatting control, larger JS bundle, inconsistent across browsers.

> **Decision: Backend Export (Option A).** Server-side generation with Python libraries gives us much better control over document formatting and consistent output.

## 3. Export Endpoint — Detailed Specification

### `POST /api/export`

**Request:**
```json
{
  "content": "# Summary: Introduction to Biology\n\n## Cell Structure\n- Cells are...",
  "format": "pdf",
  "title": "Biology Chapter 5 - Summary",
  "type": "summary"
}
```

| Field | Type | Required | Validation |
|---|---|---|---|
| `content` | `str` | Yes | Min 1 char, max 200,000 chars |
| `format` | `str` | Yes | Must be `pdf` or `docx` |
| `title` | `str` | No | Default: "Study Aid", max 200 chars |
| `type` | `str` | No | One of: summary, key_terms, quiz, study_guide. Used for formatting hints |

**Response (Success — 200):**
- Content-Type: `application/pdf` or `application/vnd.openxmlformats-officedocument.wordprocessingml.document`
- Content-Disposition: `attachment; filename="biology-ch5-summary.pdf"`
- Body: Binary file data

**Response (Error — 400):**
```json
{
  "error": "ExportError",
  "detail": "Content is empty. Nothing to export."
}
```

## 4. PDF Generation

### Library: fpdf2

`fpdf2` is a lightweight, pure-Python PDF generation library. It's the successor to `fpdf` and supports Unicode, images, and tables.

### PDF Document Structure

```
┌────────────────────────────────┐
│          HEADER                 │
│  StudyForge | [Study Aid Type] │
│  Generated: 2026-05-19         │
├────────────────────────────────┤
│                                │
│  Title (Large, Bold)           │
│  ────────────────────          │
│                                │
│  Section Heading               │
│  • Bullet point text           │
│  • Bullet point text           │
│    • Sub-bullet point          │
│                                │
│  ┌──────────────────────────┐  │
│  │ Term    │ Definition     │  │
│  │─────────│────────────────│  │
│  │ Cell    │ Basic unit of  │  │
│  │         │ life           │  │
│  └──────────────────────────┘  │
│                                │
├────────────────────────────────┤
│          FOOTER                 │
│  Page 1 of 3                   │
│  Generated by StudyForge       │
└────────────────────────────────┘
```

### Markdown to PDF Conversion

The core challenge is converting Markdown to a formatted PDF. This requires parsing the Markdown and mapping elements to PDF drawing commands.

### Markdown Element to PDF Mapping

| Markdown Element | PDF Rendering |
|---|---|
| `# Heading 1` | Font size 20, bold, blue color, bottom border |
| `## Heading 2` | Font size 16, bold, dark gray, top margin |
| `### Heading 3` | Font size 13, bold, normal color |
| `- Bullet point` | Indented with "•" character, regular font |
| `  - Sub-bullet` | Double indented with "◦" character |
| `**Bold text**` | Bold font weight |
| `*Italic text*` | Italic font style |
| `| Table |` | Bordered table with alternating row colors |
| `---` (horizontal rule) | Thin gray line across the page |
| `> Blockquote` | Indented, italic, left border |
| `` `code` `` | Monospace font, light gray background |
| `- [ ] Task` | Checkbox symbol (☐) before text |
| `- [x] Task` | Checked symbol (☑) before text |

### Markdown Parser for PDF

Create a `MarkdownToPDF` converter class that:

1. Splits content by lines
2. Identifies each line's Markdown type (heading, bullet, table, etc.)
3. Applies appropriate PDF formatting
4. Handles page breaks (don't split a table row across pages)

### Font Configuration

```
Title font:    Helvetica Bold, 20pt
Heading 2:     Helvetica Bold, 16pt
Heading 3:     Helvetica Bold, 13pt
Body text:     Helvetica, 11pt
Bullet text:   Helvetica, 11pt
Code text:     Courier, 10pt
```

> **Unicode support:** fpdf2 supports Unicode via TrueType fonts. If you need emoji (for study guides with 💡🔗❌✅), you'll need to include a font that supports them, like **Noto Emoji**. For MVP, replace emojis with text equivalents in the PDF.

### Color Scheme

```
Title:           RGB(31, 41, 55)     — Dark charcoal
Headings:        RGB(37, 99, 235)    — Blue
Body text:       RGB(55, 65, 81)     — Dark gray
Table header:    RGB(37, 99, 235)    — Blue background, white text
Table rows:      Alternating RGB(249, 250, 251) and white
Horizontal rule: RGB(209, 213, 219)  — Light gray
```

## 5. Word (DOCX) Generation

### Library: python-docx

`python-docx` is the standard Python library for creating Word documents.

### Document Structure

Similar to the PDF, but using Word's native formatting:

```
┌────────────────────────────────┐
│  StudyForge — [Type]           │  ← Custom header
│  Generated: 2026-05-19         │
├────────────────────────────────┤
│                                │
│  Title (Heading 1 style)       │
│                                │
│  Section (Heading 2 style)     │
│  • Bullet list                 │
│  • Bullet list                 │
│    • Nested bullet             │
│                                │
│  Table with styled cells       │
│                                │
├────────────────────────────────┤
│  Page 1 of 3 — StudyForge     │  ← Custom footer
└────────────────────────────────┘
```

### Markdown to DOCX Conversion

| Markdown Element | DOCX Element |
|---|---|
| `# Heading 1` | `document.add_heading(text, level=1)` |
| `## Heading 2` | `document.add_heading(text, level=2)` |
| `### Heading 3` | `document.add_heading(text, level=3)` |
| `- Bullet` | `document.add_paragraph(text, style='List Bullet')` |
| `  - Sub-bullet` | `document.add_paragraph(text, style='List Bullet 2')` |
| `**Bold**` | `run.bold = True` |
| `*Italic*` | `run.italic = True` |
| `| Table |` | `document.add_table()` with styled cells |
| `---` | `document.add_paragraph().add_run().add_break()` or a thin border |
| `> Blockquote` | Custom paragraph style with indentation |
| `- [ ] Task` | `List Bullet` with "☐" prefix |
| `- [x] Task` | `List Bullet` with "☑" prefix |

### Custom Styles

Define custom Word styles for consistent formatting:

```
Style: "StudyForge Title"     → Heading 1, 20pt, blue
Style: "StudyForge Heading"   → Heading 2, 14pt, dark blue
Style: "StudyForge Body"      → Normal, 11pt, dark gray
Style: "StudyForge Bullet"    → List Bullet, 11pt
Style: "StudyForge Table"     → Table Grid, with header row styling
```

### Advantages of DOCX over PDF

- Users can **edit** the exported document
- Better for students who want to annotate or add their own notes
- Native table support with resizing
- Easy to change fonts, colors, etc. after export

## 6. Export Service Architecture

### `services/exporter.py`

```
ExportService
├── export(content, format, title, type) → bytes
│   ├── _parse_markdown(content) → list of MarkdownElements
│   ├── _export_pdf(elements, title, type) → bytes
│   └── _export_docx(elements, title, type) → bytes
```

### MarkdownElement Data Structure

Create a simple parser that converts Markdown lines into structured elements:

```
MarkdownElement:
    type: "heading1" | "heading2" | "heading3" | "paragraph" | "bullet" |
          "sub_bullet" | "table_row" | "table_header" | "horizontal_rule" |
          "blockquote" | "checkbox" | "checkbox_checked"
    content: str          # The text content (Markdown formatting stripped)
    bold_ranges: list     # Positions of bold text
    italic_ranges: list   # Positions of italic text
    raw: str              # Original Markdown line
```

### The Markdown Parsing Challenge

Parsing Markdown perfectly is complex. For the MVP, focus on the most common elements:

**Must support:**
- Headings (H1, H2, H3)
- Bullet points (single and nested)
- Bold and italic text
- Tables (simple pipe-delimited)
- Horizontal rules
- Task lists / checkboxes

**Nice to have:**
- Numbered lists
- Code blocks
- Blockquotes

**Not needed for MVP:**
- Images
- Links (render as plain text)
- Complex nested structures

## 7. File Response Handling

### Backend: Returning Binary Files

The export endpoint should return a `StreamingResponse` (FastAPI):

**Response headers:**
```
Content-Type: application/pdf (or application/vnd.openxmlformats-officedocument.wordprocessingml.document)
Content-Disposition: attachment; filename="study-aid-summary.pdf"
Content-Length: [file size in bytes]
```

### Filename Generation

Generate a clean filename from the title:

```
Input title: "Biology Chapter 5 — Cell Structure"
Sanitized:   "biology-chapter-5-cell-structure"
With type:   "biology-chapter-5-cell-structure-summary"
Final:       "biology-chapter-5-cell-structure-summary.pdf"
```

Sanitization rules:
- Convert to lowercase
- Replace spaces and special characters with hyphens
- Remove consecutive hyphens
- Truncate to 100 characters
- Append study aid type
- Append file extension

### Frontend: Triggering Download

When the frontend receives the binary response, trigger a download:

```
1. Make POST request with responseType: 'blob'
2. Create a Blob URL from the response data
3. Create a temporary <a> element
4. Set href to the Blob URL
5. Set download attribute to the filename
6. Programmatically click the link
7. Revoke the Blob URL (cleanup)
```

The filename can be extracted from the `Content-Disposition` header.

## 9. Edge Cases & Error Handling

| Scenario | Handling |
|---|---|
| Very long content (>100,000 chars) | May produce a large PDF. Set a reasonable limit or paginate |
| Content with only a title | Generate a near-empty document with a message |
| Content with complex tables | Best-effort rendering; very wide tables may overflow |
| Unicode characters | fpdf2 handles Unicode with TTF fonts; python-docx handles natively |
| Emoji in content | Replace with text descriptions in PDF; Word handles most emoji |
| Content with code blocks | Use monospace font, gray background |
| Export fails mid-generation | Catch exceptions, return error to frontend |
| Very large tables | May span multiple pages; handle page breaks gracefully |
| Network timeout during export | Frontend shows "Export took too long. Try exporting a shorter section." |
